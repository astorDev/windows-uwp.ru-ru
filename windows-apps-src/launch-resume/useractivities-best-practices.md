# <a name="user-activities-best-practices"></a>Рекомендации для пользователя

Это руководство описывает рекомендации по созданию и обновлению действий пользователей. Обзор компонента действий пользователей в Windows, см. в разделе [по-прежнему действий пользователя, даже на устройствах](https://docs.microsoft.com/windows/uwp/launch-resume/useractivities). См. в разделе [разделе действий пользователя](https://docs.microsoft.com/windows/project-rome/user-activities/) из проекта рим для реализации действий на других платформах разработки.

## <a name="when-to-create-or-update-user-activities"></a>Когда следует создать или обновить действий пользователей

Поскольку каждое приложение отличается, он работает для каждого разработчика, чтобы определить лучший способ сопоставления действий в приложении для действий пользователей. В Cortana и временной шкалы, которые ориентированы на пользователей увеличение продуктивности и эффективности работы, помогая им вернуться к содержимому, они посещений в прошлом характеристикой действиях пользователей.

**Общие рекомендации**

* **Запишите одно действие для группы действий с соответствующим пользователем.** Это особенно актуально для списков воспроизведения или телепередачи: одно действие может обновляться с регулярными интервалами, чтобы отразить ход реализации пользовательских. В этом случае вы получите одно действие пользователя с несколькими пунктами журнал, представляющих периоды взаимодействия нескольких дней или недель. Это же относится к действий на основе документа, на которых пользователь делает постепенное хода выполнения в приложении.
* **Store пользовательских данных в облаке.** Если требуется для поддержки действий между устройствами, необходимо убедиться, что содержимое, необходимое для включения этого действия хранится в Облачное расположение. Действия для конкретного устройства будет отображаться на временной шкале на устройстве место создания действия, но может не отображаться на других устройствах.
* **Не создавать действия для действий, которые пользователям не нужно возобновить.** Если приложение используется для выполнения простой, одноразовый операций, которые не сохраняются состояния, вы, вероятно, не обязательно для создания пользовательских действий.
* **Не создавать действия для действий, выполненных другим пользователям.** Если является внешней учетной записью отправляет пользователю сообщение или @-mentions их в приложении, не следует создавать действия для этого. Этот тип действия лучше обслуживается центра уведомлений.
  * Сценарии совместной работы являются исключением: Если несколько пользователей работают на том же действии друг с другом (например, документ Word), будет существовать случаи, когда другой пользователь внес изменения после вашего пользователя. В этом случае может потребоваться обновить существующее действие, чтобы отразить изменения, внесенные в документ. Это будет включать в себя обновление существующего содержимого данные о действиях пользователя без создания нового элемента истории.

**Рекомендации для определенных типов приложений**

Несмотря на то разных каждое приложение, большинство приложений относятся к одному из следующих шаблонов взаимодействия.
* **Приложения на основе документа** — создать одно действие в документе, с помощью одного или нескольких элементов журнала, отражая периоды использования. Важно обновить ваше действие, при внесении изменений в документ.
* **Игры** — создать одно действие для каждой игры сохранения или всему миру. Если игры поддерживает только одну последовательность из уровней, можно повторно опубликовать и то же действие со временем, несмотря на то, что вы можете обновления содержимого данных, чтобы отобразить последние хода выполнения или достижения.
* **Программа приложения** — Если нет ничего в приложении, пользователям нужно оставить и возобновления, необязательно использовать действия пользователя. Хороший пример представляет собой простое приложение, например калькулятор.
* **Бизнес приложений** — существует множество приложений для управления простых задач или рабочих процессов. Создать одно действие для каждой отдельной рабочего процесса, через приложение (например, отчеты о расходах будет ли каждого отдельного действия, таким образом, пользователь может затем щелкните действие, чтобы см. в разделе, если определенный отчет был утвержден).
* **Приложения воспроизведения мультимедиа** — создать одно действие на логическое группирование содержимого (например, список воспроизведения, программы или автономного содержимого). Основной вопрос для разработчиков приложений является ли каждый фрагмент содержимого (серии Телепередачи, song) считается автономное содержимое или часть коллекции. Как правило если пользователь предпочитает воспроизведения коллекции или последовательных содержимое коллекции в целом является действие. Если они делают Выбор играть одну часть содержимого, что один фрагмент содержимого является действие. См. в разделе ниже более конкретные рекомендации.
  * **Музыка: Альбом/исполнителя/жанр** — Если пользователь выбирает альбом, исполнителя, или Жанр и попаданий **воспроизведение**, этой коллекции является действием; не следует создавать отдельно для каждой песни. Для короткого коллекции как одного альбома или воспроизводимого обратно в случайном порядке не может потребоваться обновление действия в соответствии с текущей позиции пользователя. Для долго последовательных воспроизведения например альбом или список воспроизведения записи свое местоположение в альбоме может оказаться целесообразным.
  * **Музыка: смарт-списки воспроизведения** — приложения, которые играть музыку в случайном порядке следует записать одно действие для этого списка воспроизведения. Если пользователь играет список воспроизведения во второй раз, необходимо создать дополнительные данные журнала записей для одного действия. Запись пользователя текущей позиции в списке воспроизведения не требуется, поскольку порядок имеет случайный характер.
  * **Серию Телевизионных** — Если приложение настроено для воспроизведения следующей серии после завершения текущего, следует писать только одно действие для ряда TV. При воспроизведении различных серий для нескольких сеансов просмотра вы обновите действие в соответствии с текущей позиции в серии, и будет создаваться несколько записей журнала.
  * **Фильм** — фильм является отдельным содержимым, а также должен иметь собственную запись журнала. Если пользователь останавливает, просмотре фильмов полпути, желательно записать их положение. При они хотят продолжить ее в будущем, действие может возобновить фильм, он остановился или даже запрашивать у пользователя, если они хотят возобновить или с начала.

## <a name="user-activity-design"></a>Конструктор действия пользователя

Действия пользователя состоит из трех компонентов: активация URI, визуальные данные и метаданные содержимого.
* Активация URI — это URI, который может быть передан в приложение или интерфейс, чтобы возобновить работу приложения с определенным контекстом. Как правило эти ссылки ведут обработчики протоколов для схемы (например, «my-app://page2?action=edit»). Это разработчик должен определить, как параметры URI будет обрабатываться в соответствии с их приложением. См. в разделе [активации обработки URI](https://docs.microsoft.com/windows/uwp/launch-resume/handle-uri-activation) Дополнительные сведения.
* Визуальных данных, состоящий из набор обязательных и необязательных свойств (например: название, описание или элементы инструмент Adaptive Cards), позволяет пользователю визуально определять действия. Ниже приведены рекомендации по созданию инструмент Adaptive Cards визуальные элементы для действия.
* Метаданные содержимого — данные JSON, который может использоваться для группировки и получения действия в определенном контексте. Как правило, это принимает форму http://schema.org данных. Ниже приведены рекомендации по разработке этих данных.

### <a name="adaptive-card-design-guidelines"></a>Адаптивная рекомендации по проектированию карты

Действия отображаются на временной шкале, отображаются с помощью [framework инструмент Adaptive Cards](https://docs.microsoft.com/adaptive-cards/). Если разработчик не предоставляет инструмент Adaptive Cards для каждого действия, временная шкала автоматически создает простой карты, на основе имени приложения и значков, обязательное поле заголовка и поле необязательное описание. 

Разработчики приложений, рекомендуется предоставить пользовательские карты, используя простую схему JSON адаптивной карты. См. в разделе [документации адаптивной карты](https://docs.microsoft.com/adaptive-cards/authoring-cards/getting-started) технические инструкции для создания объектов инструмент Adaptive cards. См. рекомендации ниже по разработке адаптивной карточек на действия пользователя.
* Использование образов
  * По возможности используйте уникальный образ для каждого действия. Имя приложения и значок будет автоматически отображаться рядом с картой ваше действие, а Дополнительные образы поможет пользователям найти действия, они ищут.
  * Изображения не должен содержать текст, пользователь должен для чтения. Этот текст не будет доступен для пользователей с физическими нарушениями и не удается выполнить поиск.
  * Если образ не содержит текста и может быть до размера о соотношении 2:1, вам следует использовать как фоновое изображение. Это приводит к полужирным действия карточки, которой будет выделяться на временной шкале. Изображение будет немного Затемняемый для обеспечения видна на карте, и рекомендуется только в этом случае использовать имя действия, как меньшего размера текста может быть трудно читать.
  * Если изображение не может быть обрезанное до размера 2:1, их следует поместить в карточке действия.  
    * Если пропорции квадратные или книжной ориентации, привязать изображение в правой части карточки без полей.
    * Если пропорции Альбомная, привязать изображение в правом верхнем углу карточки.
* Каждое действие требуется для предоставления имя действия, которая всегда должно отображаться.
  * Это имя должно отображаться в левом верхнем углу картой, используя параметр больших полужирным шрифтом. Очень важно, что имя указано легко запоминающееся, как это единственное часть пользователи будут видеть, когда действие отображается в сценариях Cortana. Отображение тем же именем на временной шкале облегчает пользователям просматривать большое количество действий.
* Используйте тот же стиль visual для всех действий из приложения, таким образом, чтобы пользователям можно легко найти действий приложения на временной шкале.
  * Например действия всех должен использоваться тот же цвет фона.
* Используйте сведения дополнительный текст только в случае необходимости. 
  * Во избежание переполнения карточки с текстом и использовать только дополнительную информацию, которая помогает пользователям поиск нужных действий или отражает сведения о состоянии (например, текущий ход выполнения в определенной задачи).

### <a name="content-metadata-guidelines"></a>Рекомендации по метаданные содержимого

Действия пользователя также могут содержать метаданные содержимого, который Windows и Cortana использовать для обозначения действий и формирования выводов. Действия затем могут быть сгруппированы по определенной теме, например расположение (если пользователь работает над устранением отпуска), объект (если пользователь является исследование) или действия (если пользователь хочет купить конкретного продукта в различных приложениях и веб-сайтов). Рекомендуется для представления как существительные, так и команд, участвующих в действии. 

В следующем примере метаданные содержимого JSON, следуя стандартов [Schema.org](https://schema.org/), случай: «Джон участвуют почему-то сердиты птиц Стив.»

```json
// John played angry birds with Steve.
{
  "@context": "http://schema.org",
  "@type": "PlayAction",
  "agent": {
    "@type": "Person",
    "name": "John"
  },
  "object": {
    "@type": "MobileApplication",
    "name": "Angry Birds."
  },
  "participant": {
    "@type": "Person",
    "name": "Steve"
  }
}
```
