# <a name="user-activities-best-practices"></a>Рекомендации по обеспечению действий пользователя

В этом руководстве описываются рекомендации по созданию и обновление работу пользователей. Обзор компонента действия пользователя в Windows в разделе [Действие пользователя продолжить, даже по устройствам](https://docs.microsoft.com/windows/uwp/launch-resume/useractivities). Или, содержатся в [разделе действия пользователя](https://docs.microsoft.com/windows/project-rome/user-activities/) рим проекта для реализации операций на другие платформы разработки.

## <a name="when-to-create-or-update-user-activities"></a>Когда следует создать или обновить действия пользователя

Поскольку каждое приложение отличается, — каждого разработчику, чтобы определить лучший способ сопоставить действия в приложении на действия пользователя. Действий пользователя будет демонстрировать в Cortana и временной шкалы, которые сосредоточены на производительность повышающий пользователей и эффективность им вернуться к контенту, они посещений в прошлом.

**Общие рекомендации**

* **Запишите одно действие для группы действий, связанных с ними пользователей.** Это особенно важно для воспроизведения музыки или показывает TV: одно действие могут быть обновлены через регулярные интервалы времени в соответствии с пользователя о ходе выполнения. В этом случае будут иметь одно действие пользователя с помощью нескольких элементов журнала, представляющее периоды сотрудничества через несколько дней и даже недель. То же самое применяется к действий на основе документов, в которых пользователь выбирает постепенное о ходе выполнения в вашем приложении.
* **Хранение данных пользователей в облаке.** Если необходимо обеспечить поддержку нескольких устройств действий необходимо убедитесь, что в облаке расположение будет храниться содержимое, необходимые для включения этого действия. Действия определенных устройств будет отображаться на шкале на устройстве место создания операции, но может не отображаться на других устройствах.
* **Не создавайте действий для действия, которые пользователи не требуется возобновить.** При использовании приложения для выполнения простых, одноразовый операций, которые не сохраняет состояние, вероятно, не нужно создать действие пользователя.
* **Не создавайте действий для действия выполнены другими пользователями.** Если внешний учетной записи будет отправлять пользователя или @-mentions их в вашем приложении не следует создавать действия для этого. Этот тип действия упрощает решение с помощью уведомлений центр действие.
  * Сценарии совместной работы, исключение: Если несколько пользователей работают на одном активности вместе (например, документ Word), будут случаев, когда другой пользователь внес изменения после пользователя. В этом случае необходимо обновить существующего действия для отражения изменений, внесенных в документ. Это может включать в себя обновление существующих данных контента активность пользователей без создания нового элемента журнала.

**Рекомендации для конкретных типов приложений**

Во время каждого приложения отличается, большинство приложений делятся на следующие шаблоны взаимодействия.
* **Приложения на основе документов** — создать одно действие каждого документа, с помощью одного или нескольких элементов журнала, что отражает периоды использовать. Важно обновить действие при внесении изменений в документ.
* **Игры** — создать одно действие для каждой игры сохранить или world. Если игра поддерживает только один последовательность уровней, можно повторно опубликовать того же действия со временем, несмотря на то, что вы можете обновлять данные контента для отображения последних хода выполнения или достижения.
* **Служебная программа приложения** — нет ничего в вашем приложении, пользователи должны оставьте и возобновления, не требуется ли использовать действия пользователя. Хороший пример — это простое приложение как калькулятор.
* **Бизнес - приложений** , существует много приложений для управления простых задач или рабочих процессов. Создать одно действие для каждой отдельной рабочего процесса, через приложения (например, отчеты о расходах будет ли каждого отдельного действия, чтобы пользователь может нажмите кнопку действия для просмотра, если утверждения конкретного отчета).
* **Воспроизведение мультимедиа приложения** — создать одно действие на логической группировки контента (например, содержимое списка воспроизведения, программы или автономного). Вложенный вопрос для разработчиков приложений является ли каждая часть содержимого (TV серии, песни) считается отдельного содержимого или частью коллекции. Как правило если пользователь предпочитает воспроизведение семейства сайтов или последовательного контента семейства сайтов в целом — это действие. Если они необязательно для воспроизведения одну часть содержимого, одной части содержимого является действие. В разделе более конкретные инструкции ниже.
  * **Музыки: альбом, исполнителя/жанр** , если пользователь выбирает альбом, исполнителя или Жанр и попадает **воспроизведения**, этой коллекции — это действия; не записывать отдельно для каждого песни. Для короткий коллекций как одного альбома или семейств сайтов воспроизведения в случайном порядке не может потребоваться обновить активности в соответствии с текущей позиции пользователя. Срок последовательного проигрывания например альбом или список воспроизведения записи позицию в альбом смысл.
  * **Музыки: смарт-списки** — приложения, которые воспроизведения музыки в случайном порядке, должны записать одно действие для этого списка воспроизведения. При воспроизведении списка воспроизведения еще раз, необходимо создать дополнительные журнал записей для одного действия. Запись пользователя текущую позицию в списке воспроизведения нет необходимости, так как порядок в случайном порядке.
  * **Цикл передач** — Если ваше приложение настроена для воспроизведения Следующая передача после завершения текущего, указывайте одно действие для цикл передач. При воспроизведении различных эпизодов между несколькими сеансами просмотра будут обновляться действие в соответствии с текущей позиции в серии, и создается несколько записей журнала.
  * **Фильм** — фильма является одну часть содержимого и должен иметь собственную запись журнала. Если пользователь останавливает просмотра как часть фильма через, рекомендуется записать их расположения. При возможным для его возобновления в будущем, действие может возобновить фильмов, где она была прервана или даже запрашивать у пользователя, если они хотят возобновить или в начале.

## <a name="user-activity-design"></a>Разработка активности пользователя

Действия пользователя состоят из трех компонентов: активация URI, визуальных данных и метаданных контента.
* Активация URI — это URI, который может быть передан приложению или взаимодействия для восстановления приложения с помощью определенного контекста. Как правило эти ссылки указывают обработчики протоколов для схемы (например, «my-app://page2?action=edit»). Это разработчик должен определить, как будут обрабатываться параметры URI свои приложения. Для получения дополнительных сведений в разделе [Обработка URI активации](https://docs.microsoft.com/windows/uwp/launch-resume/handle-uri-activation) .
* Visual данных, состоящая из набора свойств, обязательные и Дополнительные (например: заголовок, описание или элементы адаптивный карточки), позволяют пользователям визуально Определите действие. Ниже приведены инструкции по созданию визуальных элементов адаптивный карточку для действий пользователя.
* Метаданные контента является данных JSON, который можно использовать для группировки и получения действия в указанном контексте. Как правило, это имеет форму http://schema.org данных. Ниже приведены рекомендации по заполнении эти данные.

### <a name="adaptive-card-design-guidelines"></a>Рекомендации по проектированию адаптивный карточки

Действия отображаются в временной шкалы, отображаются с помощью [framework адаптивный карточки](https://docs.microsoft.com/adaptive-cards/). Если разработчик не предоставляет адаптивный карточку для каждого действия, временная шкала автоматически создаст простой карточки на основе имени/значок приложения, обязательные поля заголовка и необязательное поле Описание. 

Разработчики приложений, рекомендуется для предоставления настраиваемых карты с помощью простой схемы адаптивный карточки JSON. Обратитесь к [документации адаптивный карточек](https://docs.microsoft.com/adaptive-cards/authoring-cards/getting-started) технические инструкции по инструкции для создания объектов адаптивный карточки. Обратитесь к рекомендаций по проектированию адаптивный карточек в работу пользователей.
* Использование изображений
  * Используйте уникальное изображение для каждого действия, если это возможно. Имя приложения и значок автоматически отображается рядом с элементом карточки действий пользователя; Дополнительные изображения поможет пользователям найти действие для поиска.
  * Изображения не должен содержать текст, который пользователь должен прочитать. Текст не будут доступны пользователям, имеющим предоставления специальных возможностей, не удается выполнить поиск.
  * Если изображение не содержит текст и можно обрезки для о соотношение 2:1, вам следует использовать как фонового изображения. Это приводит к полужирным шрифтом активности карточки, которой будет выделить на шкале. Изображение будет темнее немного для обеспечения текст отображается на экране из карточки, рекомендуется использовать только имя действия в этом случае как меньшего размера текста может быть трудно читать.
  * Если изображение не удается обрезается до 2:1, вы должны находиться в пределах карточки активности.  
    * Если пропорции квадратные или книжную, привязать изображение в правой части карточки без полей.
    * Если пропорции — это среда, привязки изображения в верхнем правом углу карточки.
* Каждого действия необходимо предоставить имя действия, который всегда должны быть отображены.
  * Это имя должно отображаться в левом верхнем углу карточки с помощью параметра больших полужирным шрифтом. Не важно, что имя введено опознать, как это единственный часть будет выводиться для пользователей при отображении действие в сценарии Cortana. Отображение таким же именем в временной шкалы облегчает пользователям просмотр большое число действий.
* Используйте тот же visual стиль для всех действий из вашего приложения, чтобы пользователи легко могут находить ваше приложение действий на шкале.
  * Например действия всех должен использоваться один и тот же цвет фона.
* Используйте сведения дополнительный текст только в случае необходимости. 
  * Избежать заполнения карточки текст и использовать только дополнительную информацию, которая помогает пользователям в поиске правом активности или отражает сведения о состоянии (например, текущих в конкретную задачу).

### <a name="content-metadata-guidelines"></a>Рекомендации по контента метаданных

Действия пользователя также может содержать контента метаданные, которые Windows и Cortana использовать для обозначения действий и создания вывод. Операции можно сгруппировать затем по определенной теме, такие как расположение (если пользователь является исследование отпусках), object (если пользователь является исследование) или действие (если пользователь покупок для определенного продукта в различных приложений и веб-сайтов). Рекомендуется представляют существительных и команд, участвующих в действии. 

В следующем примере контента метаданных JSON, выполнив стандартов [Schema.org](https://schema.org/)представляет сценария: «John участвуют Сердитая птицы Стив.»

```json
// John played angry birds with Steve.
{
  "@context": "http://schema.org",
  "@type": "PlayAction",
  "agent": {
    "@type": "Person",
    "name": "John"
  },
  "object": {
    "@type": "MobileApplication",
    "name": "Angry Birds."
  },
  "participant": {
    "@type": "Person",
    "name": "Steve"
  }
}
```
