# <a name="app-capability-declarations"></a>Объявления возможностей приложения

Возможности должны быть заявлены в [манифесте пакета](https://msdn.microsoft.com/library/windows/apps/BR211474) приложения UWP для получения доступа к определенным API или ресурсам, например, изображениям, музыке или устройствам, в частности, камере или микрофону.

Доступ к определенным ресурсам или API запрашивается при помощи объявления возможностей в [манифесте пакета](https://msdn.microsoft.com/library/windows/apps/BR211474) приложения. Общие возможности можно объявлять с помощью [Конструктора манифестов](packaging-uwp-apps.md#configure-an-app-package) в Microsoft Visual Studio или добавлять вручную. Подробнее: [Указание возможностей в манифесте пакета](https://msdn.microsoft.com/library/windows/apps/BR211477). Не забывайте, что когда пользователи получают ваше приложение из Магазина, они уведомляются обо всех объявленных возможностях приложения. Поэтому не стоит объявлять возможности, которые вашему приложению не требуются.

Некоторые возможности обеспечивают приложениям доступ к *конфиденциальному ресурсу*. Эти ресурсы считаются конфиденциальными, так как они могут получать доступ к личным данным пользователя или стоить пользователю денег. Настройки конфиденциальности, управляемые приложением "Параметры", позволяют пользователю динамически управлять доступом к конфиденциальным ресурсам. Таким образом, важно, чтобы приложение не считало конфиденциальные ресурсы всегда доступными. Подробнее о доступе к конфиденциальным ресурсам см. в разделе [Руководство по приложениям, учитывающим требования конфиденциальности](https://msdn.microsoft.com/library/windows/apps/Hh768223). Возможности, которые обеспечивают приложений с доступом к *конфиденциальный ресурс* отмечены звездочкой (\*) рядом с возможностью сценария.

Существует несколько видов возможности.

- [Основные возможности, — использовать](#general-use-capabilities), которые действуют для наиболее распространенных сценариев приложений.
- [Возможности устройства](#device-capabilities), что позволяет вашему приложению доступ к периферийных устройств и внутренних устройств.
- [Ограниченные возможности](#restricted-capabilities), которой требуется утверждение для отправки в Microsoft Store и (или) будут обычно доступны только для корпорации Майкрософт и некоторых партнеров.
- [Пользовательские возможности](#custom-capabilities).

## <a name="restricted-capabilities"></a>Ограниченные возможности

Если приложение объявляет никакие ограниченные возможности, необходимо предоставить сведения во время [процесс отправки приложения](../publish/app-submissions.md) требуется утвердить, чтобы опубликовать приложение в Microsoft Store. Это можно сделать на [параметров отправки](../publish/manage-submission-options.md#restricted-capabilities) страницы вашего отправления, объясняя, как ваше приложение использует каждый ограниченные возможности, которая объявляет.

> [!IMPORTANT]
> Ограниченные возможности предназначены для особых случаев. Использование этих возможностей весьма ограничено, они проверяются на соответствие требованиям политики выставления Магазина и анализируются дополнительно. Обратите внимание, вы можете загружать неопубликованные приложения, которые объявляют ограниченные возможности без необходимости получать все утверждения. Утверждение требуется только при отправке этих приложений в Store.

Не забудьте объявить, что эти ограниченные возможности, если они действительно нужны приложения. Они могут использоваться в банковских операциях с двухфакторной проверкой подлинности, а также в ситуациях, при которых пользователям выдается смарт-карта с цифровым сертификатом, подтверждающим их личность. Другие приложения могут первоначально предназначаться для корпоративных клиентов. Таким приложениям нужен доступ к корпоративным ресурсам с учетными данными пользователя домена.

Чтобы объявить функцию с ограниченным доступом, изменить ваш [манифест пакета приложения](https://msdn.microsoft.com/library/windows/apps/BR211474) исходный файл (`Package.appxmanifest`). Добавить **xmlns:rescap** объявление пространства имен XML и использовать **rescap** префикса при объявлении ограниченных возможностей. Например, вот как объявить возможность **appCaptureSettings**.

```xml
<?xml version="1.0" encoding="utf-8"?>
<Package
    ...
    xmlns:rescap="http://schemas.microsoft.com/appx/manifest/foundation/windows10/restrictedcapabilities"
    IgnorableNamespaces="... rescap">
...
<Capabilities>
    <rescap:Capability Name="appCaptureSettings"/>
</Capabilities>
</Package>
```

### <a name="restricted-capability-approval-process"></a>Процедура утверждения ограниченной возможности

Раньше для получения разрешения на использование возможности требовалось обратиться в службу поддержки. Мы теперь позволяют это сделать в [центра партнеров](https://partner.microsoft.com/dashboard/) как часть [процесс отправки](../publish/app-submissions.md).

При отправке пакетов для отправки мы определим, объявляются ли любой ограниченные возможности. В этом случае вам потребуется предоставить подробную информацию о том, как каждая из этих возможностей используется в продукте. Сделать это можно на странице [Параметры отправки](../publish/manage-submission-options.md#restricted-capabilities). Предоставьте как можно более подробную информацию, чтобы мы могли понять, почему вашему продукту необходимо объявить эту возможность. Обратите внимание, что в этом случае сертификация вашей отправки может занять несколько больше времени.

В процессе сертификации наши тест-инженеры изучат предоставленную информацию, чтобы определить утверждено ли использование соответствующей возможности в вашей отправке. Обратите внимание, что в этом случае сертификация вашей отправки может занять несколько больше времени. Если мы утвердим использование вами этой возможности, процедура сертификации вашего приложения будет продолжена. Как правило, не требуется повторять процедуру утверждения возможности при отправке обновлений в приложение (за исключением случаев, когда объявляются дополнительные возможности).

Если мы не разрешили использовать возможность, сертификацию не пройдут заявку и мы предоставим обратной связи в отчете сертификации. Можно создать новую отправку и передать пакеты, в которых не объявлена эта возможность или, если применимо, решить все проблемы, связанные с использованием этой возможности, и запросить утверждение в новой отправке.

> [!NOTE]
> Если заявку использует изолированной среды для разработки в центре партнеров (например, это происходит для любого игру, которая интегрируется с Xbox Live), необходимо запросить утверждение заранее, а не предоставляют сведения о **параметров отправки** страницы . Для этого посетите [страницу поддержки разработчиков под Windows](https://developer.microsoft.com/windows/support). Выберите раздел поддержки разработчиков **проблема мониторинга**, тип проблемы **отправки приложений**и Subcategory **других**. Затем описывают, как вы используете функцию и почему это необходимо для вашего продукта. Если вы не предоставите все необходимые сведения, ваш запрос будет отклонен. Вам также можете быть предложено предоставить дополнительные сведения. Обратите внимание, что этот процесс обычно занимает 5 рабочих дней или более, поэтому отправляйте запросы заблаговременно.
>
> Можно также использовать этот метод запроса утверждения (вместо того, чтобы указать эти сведения во время отправки), ли вы используете изолированной среды для разработки, если вы предпочитаете, чтобы убедиться, что вы утверждены для использования ограниченные возможности, прежде чем ваши Отправка.

## <a name="custom-capabilities"></a>Пользовательских возможностей

[Ограниченные возможности](#restricted-capabilities) предыдущем разделе описан процесс утверждения же возможности, можно использовать для запроса утверждения для использования пользовательских возможности. [Внедренных SIM](/uwp/api/windows.networking.networkoperators.esim) API-интерфейсы являются примерами API, которые требуют пользовательских возможностей. Если требуется только локальный запуск приложения в режиме разработчика, не требуется пользовательских возможностей. Но вам необходимо будет опубликовать приложение в Microsoft Store или запустить его за пределами режим разработчика.

Если у вас есть Windows консультанту (TAM), можно работать с помощью к менеджеру по технической ПОДДЕРЖКЕ, чтобы запросить доступ. Дополнительные сведения см. в [обратитесь к своему ТЕХНИЧЕСКОМУ Microsoft](/windows-hardware/drivers/mobilebroadband/testing-your-desktop-cosa-apn-database-submission#contact-your-microsoft-tam).

Чтобы объявить функцию пользовательских, изменить ваш [манифест пакета приложения](https://msdn.microsoft.com/library/windows/apps/BR211474) исходный файл (`Package.appxmanifest`). Добавить **xmlns:uap4** объявление пространства имен XML и использовать **uap4** префикса при объявлении пользовательских возможностей. Рассмотрим пример.

```xml
<?xml version="1.0" encoding="utf-8"?>
<Package
    ...
    xmlns:uap4="http://schemas.microsoft.com/appx/manifest/uap/windows10/4">
...
<Capabilities>
    <uap4:CustomCapability Name="CompanyName.customCapabilityName_PublisherID"/>
</Capabilities>
</Package>
```
